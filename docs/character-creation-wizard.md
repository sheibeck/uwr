# AI-assisted Character Creation Wizard — Spec

This document defines a concise specification for an AI-assisted, step-by-step character creation wizard used by Unwritten Realms. It includes the flow, fields, constraints, where the AI is used, validation rules, and acceptance criteria for each step.

## Overview

An interactive stepper walks the player through 6 explicit steps: Race → Archetype → Profession → Starting Region → Description → Name. At each step the UI waits for the player's input; the AI provides optional suggestions and flavorful text but never advances the flow without explicit player confirmation. Draft state is persisted so the player can pause/resume. Finalization validates the draft and creates the final `Characters` row.

Note: the player is only responsible for supplying the six items above (race, archetype, profession, starting region, description, and name). All other details (derived stats, starting gear, racial and profession abilities, health/mana, equipment slots, etc.) are generated by the AI during the creative process. The orchestrator must request these AI-derived fields (and validate them) before calling `finalize_character_draft`.

## High-level artifacts

- Character draft schema (Zod + JSON Schema) — draft is partial/optional fields.
- Prompt templates for each step (JSON output encouraged).
- Reducers:
  - `save_character_draft(draftId?: string, draft: CharacterDraft)` — upsert.
  - `get_character_draft(draftId?: string, ownerId?: string)` — fetch.
  - `finalize_character_draft(draftId: string)` — validate then create final character.
- Orchestrator helper to run AI prompts and validate outputs.
- Client UI: a stepper component that calls orchestrator endpoints and persists drafts via reducers.

## Wizard Steps (AI role & acceptance criteria)

Each step must require explicit player confirmation before progressing.

### Step 1 — Race
- Prompt: "Which race do you want?"
- Options: Human, Elf, Dwarf, Gnome, Halfling, Orc, Troll, Ogre, Cyclops, Half-Giant, or custom.
- AI: offers evocative descriptions; can invent a custom race with suggested bonuses/penalties.
- Constraints: listed races map to canonical bonuses; custom races require a short description and suggested bonuses.
- Acceptance: player confirms; race and race modifiers stored in draft.

### Step 2 — Archetype
- Prompt: "Pick an archetype: Fighter or Mystic"
- AI: short flavor text for each choice.
- Constraints: archetype ∈ {Fighter, Mystic}.
- Acceptance: player confirms; archetype saved.

### Step 3 — Profession (AI-generated)
- Triggered by race + archetype.
- AI must return a Profession JSON with:
  - `name` (must NOT contain literal race or archetype tokens),
  - `abilities` (1–3 items, balanced for starting play),
  - `bonuses` (attribute/resistance modifiers),
  - `starterWeapon`, `armorType`.
- Constraints: 1–3 abilities; powerful abilities reduce allowed count; names must be evocative.
- Acceptance: player confirms or requests a new suggestion; profession saved in draft.

### Step 4 — Starting Region
- Behavior: show a filtered list appropriate to race/profession; AI provides short flavor text.
- Constraints: region must be from allowed list or be a confirmed custom entry.
- Acceptance: player confirms; `startingRegion` saved.

### Step 5 — Brief Description (Appearance)
- Prompt: "Describe what another person would see."
- AI: can auto-generate or refine; limit e.g., 280 chars.
- Acceptance: player confirms and description saved.

### Step 6 — Name
- Prompt: "What’s the character name?"
- Constraints: 3–32 chars, allowed characters only; server-side uniqueness enforced.
- AI: suggests names or alternatives on conflict.
- Acceptance: player confirms a validated, unique name; saved.

### Finalize — Create Profile & Narrative Wrap-up
- Compose final fields applying modifiers:
  - Core attributes start at baseline (20) then apply race/archetype/profession modifiers.
  - Starting Health: Fighter 100 / Mystic 50 ± modifiers.
  - Starting Mana: Mystic 100 / Fighter 50 ± modifiers.
  - Level = 1, XP = 1.
  - Armor type, starter weapon, abilities, backpack/inventory starter contents.
- Validation: run Zod/JSON schema + server rules (name uniqueness, attribute ranges, profession name rule).
 - Validation:
   - The orchestrator MUST ensure AI-derived fields are present (attributes, abilities, starterWeapon, armorType, etc.) and validate them with the draft schema before calling `finalize_character_draft`.
   - Server-side checks include: name uniqueness, attribute ranges, and profession-name rules; these are defensive checks and not a substitute for orchestrator validation.
 - Acceptance:
   - Orchestrator has validated AI-derived fields and resolved any issues (or returned validation errors to the client).
   - `finalize_character_draft` creates the `Characters` row and returns `characterId`.
   - Server returns the serialized final profile and a narrative wrap-up paragraph to the client.

## Data Models

CharacterDraft (partial):
- `draftId: string`
- `ownerId: string`
- `race`, `archetype`, `professionName`, `professionAbilities[]`, `professionBonuses`, `startingRegion`, `description`, `name`, `aiMetadata`, `lastUpdated`

Characters (final) maps to existing `Characters` table: `id`, `owner_id`, `name`, `description`, `race`, `archetype`, `profession`, `starting_region`, attributes, `backpack`, equipped slots, `quests`, etc.

## Validation Rules (server-side)

- Name: regex ^[A-Za-z0-9' -]{3,32}$ and unique.
- Attributes: base 20 modified by bonuses; final values in [1, 40].
- Abilities: ≤ 280 chars each; 1–3 starting abilities.
- Profession name must not contain the race or archetype text (simple substring check).
- `save_character_draft` verifies `ownerId === ctx.sender`.

## AI Prompting Guidelines

- Use focused prompts per step and instruct the AI to return JSON that matches a tiny schema.
- System role: enforce stepwise progression and that outputs should be JSON when requested.
- Example profession prompt: produce `{name, abilities:[{name,desc}], bonuses:{str:+1,...}, starterWeapon, armorType}` and a 1–2 sentence flavor blurb.

## Orchestrator responsibilities

- Run AI prompts, parse JSON, validate with Zod schemas.
- Persist drafts via `save_character_draft`.
- Check name uniqueness and request AI alternatives if taken.
- Return structured suggestion + flavor text to client.

## Reducers / API surface

- `save_character_draft(draftId?: string, draft: CharacterDraft)` — upsert draft; returns `draftId`.
- `get_character_draft(draftId?: string, ownerId?: string)` — fetch drafts.
- `finalize_character_draft(draftId: string)` — validate and create final `Characters` row (may call `create_character` internally), returns created character object or validation errors.

## UX notes

- Require explicit confirm after any AI suggestion.
- Provide "Regenerate suggestion" button for AI-based steps.
- Auto-save drafts periodically and on each confirm step.

## Telemetry & safety

- Log AI prompts/outputs and validation results to a dev sink for tuning.
- Rate-limit AI calls per account.

## Tests / Acceptance

- Unit tests: AI JSON parsing, validation rules, reducer behaviors.
- Integration tests: full flow happy path; name conflict flow.
- Manual acceptance: run the wizard, finalize, verify character row created and narrative produced.
