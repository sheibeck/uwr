<objective>
Replace the real-time tick-based combat loop with a round-based resolution engine. Add new schema tables for round state, player actions, and combat narratives. Restructure combat.ts to use submit-then-resolve pattern instead of continuous 1-second ticks.

Purpose: This is the foundation for narrative combat -- rounds must exist before the LLM can narrate them. The existing combat_loop scheduled reducer, EffectTick, HotTick, and CastTick scheduled reducers are replaced by a single resolve_round function that processes everything in one transaction per round.

Output: Working round-based combat engine where players submit actions, timer or all-submitted triggers resolution, effects tick per-round, and mechanical messages appear instantly.
</objective>

<task type="auto">
  <name>Task 2: Rewrite combat engine from tick-based to round-based resolution</name>
  <files>spacetimedb/src/reducers/combat.ts, spacetimedb/src/helpers/combat.ts, spacetimedb/src/index.ts</files>
  <action>
This is the core engine rework. The goal is to replace the continuous combat_loop tick with a round-based submit-then-resolve pattern. Preserve ALL existing combat mechanics (damage formulas, crits, scaling, aggro, effects) but restructure the execution flow.

**In combat.ts:**

1. **Replace combat_loop scheduled reducer** with resolve_round_timer:
   - The old combat_loop fires every 1s and processes auto-attacks, enemy AI, etc.
   - New resolve_round_timer: receives { arg } with combatId and roundNumber from RoundTimerTick
   - Verify the round is still in 'action_select' state (may have been resolved early by all-submitted)
   - Fill in auto-attack defaults for any player who didn't submit an action
   - Call resolveRound(ctx, combatId, round)

2. **Add submit_combat_action reducer:**
   - Args: characterId (u64), actionType (string), abilityTemplateId (u64 optional), targetEnemyId (u64 optional), targetCharacterId (u64 optional)
   - Validate: character owned by sender, character in active combat, current round in 'action_select' state
   - Validate ability: if actionType is 'ability', verify abilityTemplateId exists in player's abilities and is not on cooldown
   - Upsert pattern: find existing CombatAction for this character+combat+round, update if exists, insert if not (allows changing action before round locks)
   - After upsert, check if ALL active participants have submitted actions
   - If all submitted: delete the RoundTimerTick for this round (cancel timer), call resolveRound inline

3. **Create resolveRound function** (the heart of the new engine):
   - Set round state to 'resolving'
   - Collect all CombatAction rows for this round
   - **Process player actions** in submission order:
     - For 'ability' actions: reuse existing executeAbility logic from helpers/combat.ts but adapted for batch mode
     - For 'auto_attack': reuse existing auto-attack damage calculation
     - For 'flee': reuse existing flee logic (roll success/fail, if success remove from combat)
     - Each action produces mechanical messages via appendPrivateEvent/logPrivateAndGroup (instant)
   - **Process enemy actions** (reuse existing enemy AI logic):
     - Standard enemies: 1 action (ability or auto-attack based on AI weights)
     - Boss enemies (template.isBoss if it exists, otherwise just 1 action): 2 actions
     - Enemy ability selection reuses existing pickEnemyAbility/AI logic
   - **Tick effects once** (replaces EffectTick, HotTick):
     - Process all CharacterEffect rows for participants: apply DoT damage, HoT healing, decrement duration
     - Process all CombatEnemyEffect rows: apply DoT damage, decrement duration
     - Remove expired effects (duration <= 0 after decrement)
     - Convert existing time-based durations to round counts: rounds = max(1, floor(durationMicros / EFFECT_ROUND_CONVERSION_MICROS))
   - **Process pending casts** (replaces CastTick):
     - Any PendingSpellCast rows that were started this round should complete
   - **Check victory/defeat:**
     - All enemies dead -> victory (reuse existing victory logic: XP, loot, combat results)
     - All players dead -> defeat (reuse existing defeat logic)
   - **Collect round events** into a structured summary object for LLM narration (plan 02 will use this)
   - Set round state to 'resolved'
   - **If combat continues:** create next CombatRound (roundNumber + 1, state: 'action_select'), schedule RoundTimerTick
   - **If combat ends:** cleanup CombatAction rows, mark encounter complete

4. **Modify start_combat/startCombatForSpawn:**
   - After creating CombatEncounter, create first CombatRound (roundNumber: 1, state: 'action_select')
   - Schedule RoundTimerTick for the first round
   - Determine timer duration: check if group (10s) or solo (6s) using groupId on encounter
   - Do NOT schedule old CombatLoopTick
   - Do NOT schedule separate EffectTick/HotTick/CastTick for this combat

5. **Adapt use_ability reducer:**
   - Instead of executing ability immediately, redirect to submit_combat_action with actionType 'ability'
   - Or, if the player is in round-based combat, treat use_ability as a convenience wrapper that calls submit_combat_action logic

6. **Adapt flee_combat reducer:**
   - Instead of setting fleeing status, redirect to submit_combat_action with actionType 'flee'

**In helpers/combat.ts:**
- Extract the core damage/heal/effect application logic into reusable functions if not already (executeAbilityAction, applyAutoAttackDamage, processEffectTick)
- Add a new helper: convertDurationToRounds(durationMicros: bigint): bigint -- uses EFFECT_ROUND_CONVERSION_MICROS
- When adding effects via addCharacterEffect/addEnemyEffect, if combat is round-based, store duration as round count instead of microseconds. Use a flag or check the combat type.

**In index.ts:**
- Register the resolve_round_timer scheduled reducer in scheduledReducers
- Remove or guard the old combat_loop, tick_effects, tick_hots, tick_casts registrations so they don't fire for round-based combats
- Import new tables (CombatRound, CombatAction, CombatNarrative, RoundTimerTick) in the schema imports

**Critical: Preserve all existing combat mechanics.** The damage formulas in combat_scaling.ts, crit calculations, aggro system, equipment bonuses, stat scaling -- all remain exactly as they are. Only the TIMING and EXECUTION ORDER changes from continuous ticks to discrete rounds.

**Effect duration conversion:** When effects are applied during round-based combat, convert their duration: rounds = max(1n, durationMicros / EFFECT_ROUND_CONVERSION_MICROS). Each round resolution decrements effect duration by 1. Effects with duration 0 after decrement are removed.

**Pull system interaction:** The existing pull system (PullState, PullTick, resolve_pull) initiates combat. Once combat starts and the first round begins, the pull system's job is done. Pull tick delays before combat starts remain unchanged. After startCombatForSpawn creates the encounter, it creates the first CombatRound.

**Loot handling:** Victory loot distribution (CombatResult, CombatLoot tables) remains unchanged -- it happens during resolveRound when victory is detected, same as before but now in the round resolution transaction.
