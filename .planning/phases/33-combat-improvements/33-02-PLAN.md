    ```
    if (activeCombatIdForCharacter(ctx, character.id)) {
      return failCombat(ctx, character, 'Already in combat');

    ```
    With: Allow the pull to proceed even if in combat. The resolve step will handle adding to the existing fight. Keep the pull state creation and scheduling unchanged.

    **In `resolve_pull` scheduled reducer (line 1014-1020):**
    This is the critical change. Currently:
    ```
    if (activeCombatIdForCharacter(ctx, character.id)) {
      if (spawn.state === 'pulling') {
        ctx.db.enemy_spawn.id.update({ ...spawn, state: 'available' });
      }
      ctx.db.pull_state.id.delete(pull.id);
      return;
    }
    ```
    Replace with mid-combat add logic:
    ```
    const existingCombatId = activeCombatIdForCharacter(ctx, character.id);
    if (existingCombatId) {
      // Mid-combat pull: add new enemies to existing combat
      const combat = ctx.db.combat_encounter.id.find(existingCombatId);
      if (combat) {
        // Get existing participants (already in combat)
        const existingParticipants = [...ctx.db.combat_participant.by_combat.filter(existingCombatId)]
          .map(p => ctx.db.character.id.find(p.characterId))
          .filter(Boolean);

        addEnemyToCombat(deps, ctx, combat, spawn, existingParticipants);

        // Log the add
        logPrivateAndGroup(ctx, character, 'combat',
          `You pull ${spawn.name} into the existing fight!`,
          `${character.name} pulls ${spawn.name} into the fight!`);

        // Clean up pull state
        ctx.db.pull_state.id.update({ ...pull, state: 'resolved', outcome: 'success' });
        return;
      }
    }
    ```

    **CRITICAL — Avoid duplicate participants (Pitfall 3 from research):**
    `addEnemyToCombat` inserts aggro entries for each participant. Since participants are already in combat, this correctly creates aggro entries for the NEW enemy vs existing characters. It does NOT create new combat_participant rows (that's done in startCombatForSpawn, not addEnemyToCombat). Verify by reading addEnemyToCombat — it only inserts aggro_entry rows per participant, not combat_participant rows.

    **Also handle the delayed-adds path:** If the pull outcome includes delayed adds (partial/failure outcomes that still add nearby social enemies), those also need the mid-combat check. The delayed add logic further in resolve_pull (around line 1160+) already uses addEnemyToCombat directly, so it should work. Verify this path.

    **For `start_combat` and `start_tracked_combat`:** These check `activeCombatIdForCharacter` for each participant and block. Keep this blocking behavior for direct combat start — mid-combat add only applies to pulls. The design intent is: pulls add to fights, but you cannot /startcombat on a new spawn while fighting.
